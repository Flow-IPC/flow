75/**
76 * Manages a config setup, intended for a single daemon process, by maintaining
77 * 1 or more set(s) of static config and dynamic config, each, via that number of `Option_set<>`-ready raw value
78 * `struct` types supplied by the user as template arguments. There is an even number of template args (or it will
79 * not compile); each successive pair contains, in order, a *static* `Value_set` followed by a *dynamic* one.
80 * It is possible, for each given pair, to use static config alone or dynamic config alone; in that case pass in
81 * Null_value_set for the unused one:
82 *
83 * ~~~
84 * // First config pair is static-only; second config pair -- for a separate application module perhaps --
85 * // has both a static and dynamic parts.
86 * flow::cfg::Config_manager<Static_config_general, Null_value_set, Static_socket_opts, Dynamic_socket_opts> ...;
87 * ~~~
88 *
89 * @see Static_config_manager for the probably-common use case when you have only static config and only one `struct`
90 * at that. Static_config_manager adapts Config_manager with a very simple API, avoiding parameter packs
91 * and any mention of various dynamic-config complexities.
92 *
93 * @todo Dynamic config support lacks crash rejection; though these features should be an incremental improvement
94 * around the existing code. By crash rejection I mean something like: config file
95 * X comes in; we rename it to X.unparsed; we try to parse it; program crashes -- versus it's fine, so we rename it
96 * X.parsed, and next time X.parsed is what we presumably-safely parse after any restart. Similarly invalid config
97 * may not cause a crash but still shouldn't be repeatedly re-parsed (etc.). Exact design TBD and will be informed
98 * by previous work on other projects (by echan and ygoldfel at least).
99 *
100 * ### When (and how) to use this ###
101 * First see the namespace flow::cfg doc header for a brief overview. If you have chosen to use flow::cfg for (at least
102 * some of) your config needs, what is mandatory to use is the class template Option_set. The choice is to use it
103 * directly (writing your own facilities around it) or to use this Config_manager to maintain a couple of `Option_set`s
104 * for you in a straightforward way. Config_manager supports that one straightforward way, and for daemon programs
105 * it may be a good choice. Otherwise, the idea behind Option_set is to be able to flexibly use them as needed
106 * (Config_manager providing one such way).
107 *
108 * Firstly, you may create a `const` (immutable) Config_manager via its constructor and then just use it to output
109 * a help message (log_help() or help_to_ostream()). This could be used with your program's `--help` option or similar,
110 * and that's it (no parsing takes place).
111 *
112 * ~~~
113 * // Example of a static-config-only Config_manager created just to log the help, and that's it.
114 * flow::cfg::Config_manager<Static_value_set, Null_value_set>
115 * (get_logger(), "cfg", &static_cfg_declare_opts, flow::cfg::null_declare_opts_func())
116 * .log_help();
117 * ~~~
118 *
119 * Orthogonally, of course, you may want to use it to parse things. In which case:
120 *
121 * Config_manager assumes the following setup:
122 * - 1 or more config sets, each applicable perhaps to a different module of your application/library/etc.
123 * For *each* config set, have 1 or both of the following. (Having neither is pointless; so do not. Officially
124 * behavior is undefined if you do this.)
125 * - One set of values, such that it is read from config source(s) once; then no value therein ever changes again.
126 * Call this type `S_value_set` (S for static).
127 * `S_value_set` must be a type suitable for Option_set as its `Value_set` template arg. See Option_set doc
128 * header. Spoiler alert: It essentially needs to be a `struct` of reasonably-copyable,
129 * reasonably-equality-comparable, stream-parseable, stream-printable scalars (possibly nested); it needs to
130 * declare its default values in a no-arg ctor; and it needs an option-declaring function that calls
131 * FLOW_CFG_OPTION_SET_DECLARE_OPTION() for each data member therein. The latter function must be passed to
132 * Config_manager ctor.
133 * - After ctor, static config stored herein will be at its default values. Call apply_static() (or
134 * apply_static_and_dynamic(); see below) to read the static values (from a file, etc.).
135 * - This `S_value_set` is accessible by reference-to-immutable accessor static_values() or by
136 * all_static_values().
137 * - static_values() and all_static_values() are essentially thread-safe, at least after apply_static(), in that
138 * static_values() will always return a reference to the same `S_value_set` and all_static_values() will
139 * always emit the same pointers, and the values within the `S_value_set`(s) shall never change.
140 * - A 2nd set of values, such that it is read from config source(s) at least once; but then potentially
141 * more times, and each time 0 or more of the values may be changed from the original setting.
142 * `D_value_set`, again, must be a type suitable for Option_set as its `Value_set`.
143 * - Initialize the first set of dynamic values by calling either apply_static_and_dynamic() (if your setup allows
144 * for initial dynamic values to be supplied in the same config source(s) (e.g., same file) as static ones)
145 * *and* then apply_dynamic(); or *only* the latter.
146 * - After that initial `apply*_dynamic()`: dynamic_values() returns a ref-counted pointer to the heap-allocated
147 * `D_value_set` canonical at that time. dynamic_values() may return a different pointer each time (and
148 * will, if a dynamic change is detected); but the values *at* a given return pointer will *never* change.
149 * all_dynamic_values() is an alternate approach but essentially the same idea.
150 * - Now, whenever you have reason to believe the dynamic config may have changed, call apply_dynamic().
151 * - apply_dynamic() (after the first one) and dynamic_values() are mutually thread-safe: they can be safely
152 * called concurrently with each other and/or themselves. Internally, an atomic pointer to `D_value_set`
153 * is stored; `apply_dynamic()` makes a single update to it -- after a potentially lengthy successful parse
154 * of config source(s) -- while dynamic_values() makes an atomic read followed by a pointer copy (and
155 * returns the copy). The ref-counted-ptr handle ensures the returned `D_value_set` survives as long
156 * as the user needs it, even if it's immediately replaced by new config within the Config_manager.
157 * - Since dynamic config can change at any time, it is up to the config user how they want to operate on the
158 * ref-counted handle obtained from dynamic_values(). If a potentially new value each time is fine, it's fine
159 * to simply do `cm.dynamic_values()->m_some_opt`. If one needs a consistent set of 2 or more values, or
160 * if one needs values to not change over time, then one can simply save `auto dyn_cfg = cm.dynamic_values()`
161 * and then access values through `dyn_cfg->` as long as consistency is desired. (Again: `*dyn_cfg` can never
162 * change, once `dyn_cfg` is returned through dynamic_values().)
163 * - You may also have hooks executed when dynamic config changes. This is a simple system: before
164 * any relevant `apply*_dynamic()` calls, call register_dynamic_change_listener() for each module that
165 * wants to be informed. Each callback so registered will be *synchronously* executed from `apply*_dynamic()`,
166 * when a relevant change is detected.
167 *
168 * To summarize: This is the assumed order of API calls on a given `*this`; other orders may lead to undefined behavior:
169 * - Config_manager ctor.
170 * - apply_static() or (if using dynamic config, but even so still optional) apply_static_and_dynamic().
171 * - If using dynamic config: apply_dynamic().
172 * - Simultaneously/at various times (thread-safe w/r/t each other):
173 * - static_values() (always returns a reference to the same immutable `S_value_set`).
174 * - all_static_values() (always emits the same pointers to immutable `S_value_set`s).
175 * - If using dynamic config: dynamic_values() (if returned `x`, `*x` is immutable; but may return different `x`es
176 * over time). Alternatively: all_dynamic_values().
177 * - apply_dynamic() (changes ptr returned by dynamic_values() and all_dynamic_values()).
178 * - Config-using modules' destructors/deinitialization code.
179 * - For each such module: Obviously it must run after the last `*ic_values()` call.
180 * - For each such module: It *must* run before Config_manager dtor.
181 * - Config_manager dtor.
182 *
183 * In addition, between Config_manager ctor and dtor:
184 * - (Optional) If using dynamic config: register_dynamic_change_listener() (for each module interested in it);
185 * not thread-safe against concurrent apply_dynamic(), apply_static_and_dynamic(), or
186 * unregister_dynamic_change_listener().
187 * - Functions `register_dynamic_change_listener()`ed earlier may execute synchronously from within
188 * apply_dynamic() and/or apply_static_and_dynamic().
189 * - Such user functions may load async work elsewhere.
190 * - Ensure that thing onto which they load async work exists! See register_dynamic_change_listener() for
191 * useful patterns to ensure this.
192 * - (Optional) If a callback has been registered: unregister_dynamic_change_listener(); not thread-safe against
193 * concurrent apply_dynamic(), apply_static_and_dynamic(), or unregister_dynamic_change_listener().
194 * - Any callback which accesses something which can be invalidated SHOULD be unregistered before that thing becomes
195 * invalid. If there is a chance that apply_dynamic() or apply_static_and_dynamic() might be called after it
196 * has become invalid, then the callback MUST be unregistered (otherwise, undefined behavior could be encountered
197 * when the invalid thing is accessed).
198 *
199 * ### Advanced feature: multi-source parsing and source skipping ###
200 * By default this feature is not in play, as the `bool commit` arg to apply_static_and_dynamic(), apply_static(),
201 * and apply_dynamic() defaults to `true`. The multi-source feature is enabled by the user setting it to `false`
202 * for some calls to `apply_*()`. To explain how it works consider one particular `Value_set` and an attempt
203 * to execute one of `apply_*()` methods w/r/t that `Value_set` (among potentially others to which the same
204 * discussion applies equally). For purposes of discussion assume it is a dynamic `Value_set`, and the operation
205 * in question is apply_dynamic(); the same points apply to static ones and the other two `apply_*()` calls
206 * except where noted below.
207 *
208 * Suppose the canonical (as returned by dynamic_values() or all_dynamic_values()) `Value_set` is in state B,
209 * the baseline state. (If `Value_set` is static, then B is the default state from default-cted `Value_set`, and
210 * the canonical-state accessors are static_values() and all_static_values().)
211 * Normally a single *update* consists of just one call to `apply_dynamic(commit = true)`:
212 * -# `R = apply_dynamic(P, F, true)`, where `P` is a file path, and `F` is your Final_validator_func::Type function.
213 *
214 * If `F() == Final_validator_outcome::S_FAIL`, then `R == false`; state stays at B.
215 * If it's `F() == S_ACCEPT` (and the individual options were all fine), then `R == true`; state becomes B';
216 * and if B does not equal B' (a change was detected), then the dynamic changes listener(s)
217 * (from register_dynamic_change_listener()) are synchronously called before apply_dynamic() returns.
218 * (That listener stuff does not apply to static `Value_set`s in apply_static() and apply_static_and_dynamic().)
219 *
220 * That's the default behavior without engaging this feature. To engage the feature, one has to
221 * call `apply_dynamic(commit = false)` at least once. A single *update* now consists of 2+ calls
222 * to `apply_dynamic()`:
223 * -# `R = apply_dynamic(P1, F, false)`; if `!R`, update failed/exit algorithm; else:
224 * -# `R = apply_dynamic(P2, F, false)`; if `!R`, update failed/exit algorithm; else:
225 * -# ...
226 * -# `R = apply_dynamic(Pn, F, true)`; if `!R`, update failed/exit algorithm; else done.
227 * The `true` arg indicates the final call.
228 *
229 * (`F` can be different each time, though informally we suspect that would be unorthodox. It is allowed formally.)
230 *
231 * Each apply_dynamic() call builds up a *candidate* `Value_set C` which is created (at the top of call 1)
232 * to equal baseline/initial state B and then potentially incrementally morphed by each apply_dynamic() call.
233 * Assuming no error (from individual validator or an `F() == S_FAIL`) at any stage, the effect on the *candidate*
234 * `Value_set C` is as follows:
235 * -# If `F(Cnext) == S_ACCEPT`: Modify the *candidate* by overwriting it: `C = Cnext;`.
236 * -# If `F(Cnext) == S_SKIP`: Keep the *candidate* `Value_set C` unchanged from its current value at entry to that
237 * apply_dynamic() call. `Cnext` is discarded (so the incremental update to the candidate is *skipped*).
238 *
239 * However, and this is the key:
240 * - `apply_dynamic(..., false)` does *not* modify the canonical (as returned by all_dynamic_values(), etc.)
241 * state. It remains equal to B. Hence no dynamic change listeners are invoked. However:
242 * - the final call -- `apply_dynamic(..., true)` -- makes the canonical state equal the *candidate* `Value_set C`
243 * that has been built up. Accordingly, before `apply_dynamic(..., true)` returns, if the canonical
244 * `Value_set B` differs from the candidate `Value_set C` that overwrites it, dynamic change listeners are
245 * invoked.
246 *
247 * (Hence the one-call update scenario (the vanilla one) is merely a degenerate case of the multi-call scenario;
248 * as then the last call is the only call and thus canonicalizes the candidate and calls dynamic change listeners if
249 * appropriate.)
250 *
251 * If at any stage `F(Cnext)` yields Final_validator_outcome::S_FAIL, then apply_dynamic() returns `false`. This means
252 * the entire update has failed; the *candidate* is abandoned, and you should not make the subsequent
253 * apply_dynamic() calls that may be remaining in the update. The next such call will be taken to be the 1st
254 * of another update, with a fresh candidate C created to equal B.
255 *
256 * Why use this? Why use 2+ files for a given update? Indeed you shouldn't, unless there is specifically a good
257 * reason. What might that be? The precipitating use case, and the basic one immediately obvious to the authors,
258 * is where exactly for 1 of the files the final-validator shall return Final_validator_outcome::S_ACCEPT, while
259 * for *all* others it will return Final_validator_outcome::S_SKIP. For example, if your distributed network
260 * consists of 3 sections, and each machine knows to which section it belongs, and each file is aimed at exactly 1
261 * of the 3 and thus specifies it as some option `if-section-is=` (possible values 1 or 2 or 3), then the
262 * final-validator func would:
263 * -# Check `if-section-is` option and return SKIP if the machine's actual section does not match it. Else:
264 * -# Ensure internal consistency of candidate `Value_set` (as usual).
265 * -# Return FAIL or ACCEPT depending on the answer to that (as usual).
266 *
267 * Now you've enabled conditional config in a distributed deployment.
268 *
269 * @warning In this *conditional config* use case, it is likely best to declare any given conditional option
270 * using a `_NO_ACC` variation of your `FLOW_CFG_OPTION_SET_DECLARE_OPTION*()` option of choice;
271 * that is `FLOW_CFG_OPTION_SET_DECLARE_OPTION_NO_ACC()` or similarly-postfixed variation.
272 * You probably do not want such an option's value to persist from an earlier file in the update to a later
273 * one; instead it should probably reset to its default "at the top" of each file; `_NO_ACC` variants
274 * will accomplish this.
275 *
276 * That said the use of SKIP (for all but 1) is not mandatory in such a multi-file-update setup. You can never-SKIP,
277 * or you can ACCEPT (not-SKIP) 2+ files; it is conceivable those setups have use cases as well. Informally though we
278 * reason to you as follows:
279 * - If you never SKIP, then why complicate matters by having updates consist of 2+ files in the first place?
280 * Isn't it simpler (for config deployment and other aspects of the system) to just keep it to 1?
281 * - If you ACCEPT (don't-SKIP) 2+ files, then merely looking at a given file will no longer easily show the resulting
282 * final config, as each individual update is now incremental in nature; a preceding file in the same update
283 * may have changed (versus baseline) some setting not mentioned in that particular file. This may prove to be
284 * confusing.
285 * - If you ACCEPT (don't-SKIP) exactly 1 file, then this no longer the case, and things are simple enough again.
286 * - If you ACCEPT (don't-SKIP) 2+ files, then to mitigate the problem where the final config is not quite clear
287 * just by looking at the input files, it's simple to use state_to_ostream() or log_state() to direct the
288 * result to a file or the logs, for human viewing.
289 *
290 * The above are informal recommendations for maximum simplicity or clarity, but it is entirely conceivable a more
291 * complex situation arises and requires they not be heeded.
292 *
293 * ### Relationship with the rest of your program/threads ###
294 * At this time Config_manager is deliberately simple in that it doesn't start any threads of its own; it doesn't watch
295 * file modifications to detect dynamic changes; and its methods can be called from any thread and always act
296 * entirely synchronously. (The only methods affected by thread considerations are apply_dynamic() and
297 * dynamic_values() + all_dynamic_values(); they are intentionally thread-safe w/r/t each other.)
298 *
299 * To the extent that it uses callbacks -- again, only for dynamic config needs and only optionally -- they, too, are
300 * executed synchronously. However, these callbacks can (of course) `.post()` work onto boost.asio util::Task_engine
301 * or async::Concurrent_task_loop or async::Single_thread_task_loop.
302 *
303 * Informally:
304 * It is, however, possible and perhaps probable that one should build more facilities to detect cfg changes of various
305 * kinds; for this thread(s) may be useful. We suspect it is better to build class(es) that do that and make use of
306 * Config_manager -- not expand Config_manager to do such things. Its simplicity and relative light weight and
307 * flexibility are good properties to conserve and not saddle with baggage (even optional baggage).
308 *
309 * ### Thread safety ###
310 * This is discussed in bits and pieces above and in various doc headers. Here is the overall picture however:
311 * - Separate Config_manager objects are independent of each other, so there are no thread safety issues.
312 * - For a given `*this`:
313 * - By default, no non-`const` method is safe to call concurrently with any other method. Except:
314 * - dynamic_values() / all_dynamic_values() may be called concurrently with itself/each other or any other method
315 * including `apply_dynamic(commit = *)`, as long as either `apply_dynamic(commit = true)` or
316 * `apply_static_and_dynamic(commit = true)` has succeeded at least once prior.
317 *
318 * @tparam S_d_value_set
319 * An even number (at least 2) of settings `struct`s -- see Option_set doc header for requirements for each --
320 * in order static, dynamic, static, dynamic, .... Use Null_value_set as needed; but do not use Null_value_set
321 * for both the static and dynamic parameter in a given pair (e.g., `<Null_value_set, Null_value_set, ...>`
322 * is both useless and formally disallowed).
323 *
324 * @internal
325 * ### Implementation notes ###
326 * I (ygoldfel) wrote this, originally, in 2 stages. At first it only supported exactly 2 template args: one static
327 * and one dynamic `Value_set`. In the next stage I turned it into a parameter pack, so that 4, 6, 8, ... args
328 * can be provided if desired -- to support multiple configurable modules in the application. Before this I had done
329 * some basic parameter-pack coding, but undoubtedly Config_manager is my first foray into this level of
330 * sophistication. Certainly I welcome changes to improve clarity of the implementation and other suggestions.
331 * Generally speaking the code does strike me as complex, though there's a good case to be made to the effect of:
332 * That is just the nature of the beast. It's meta-programming: having the compiler perform loops, of a sort,
333 * at *compile time*. Because `S_d_value_set...` is not homogenous -- within each adjacent pair of tparams the first
334 * and the second are *not* always treated equivalently -- the code has to be tricky at times.
335 *
336 * That said the usability of the API should not be affected by these details much. At times one has to supply
337 * additional `<template args>` -- apply_static() and apply_dynamic() come to mind in particular -- to help the
338 * compiler; but the compiler will complain about being unable to deduce the tparams, and then the next action for the
339 * user is straightforward.
340 *
341 * ### A particular trick explained ###
342 * In C++17 fold expressions are available (https://en.cppreference.com/w/cpp/language/fold), and we use them
343 * to create compile-time loops without the appalling coding/maintenance overhead of the meta-recursive pattern.
344 * (This replaces a much uglier trick involving a dummy `int[]` initializer, though that was still superior to
345 * doing the meta-recursive thing all over the place.)
346 *
347 * Suppose, say, you want to call `f<Value_set>(idx, dyn_else_st)`,
348 * where `idx` runs through [0, #S_N_VALUE_SETS), while `dyn_else_st` alternates between `false` (static) and
349 * `true` (dynamic). E.g., `f()` might parse the `idx`th slot of #m_s_d_opt_sets and do something extra if it's
350 * an odd (dynamic) slot. Note `f()` is a template parameterized on `Value_set`. Then:
351 *
352 * ~~~
353 * size_t idx = 0;
354 * bool dyn_else_st = false;
355 * (
356 * ..., // This indicates the entire (..., expr) is a fold-expression, where...
357 * (
358 * // ...the `S_d_value_set` in the following expression is the kernel of the param-pack expansion;
359 * // due to the fact the class template is: template<typename... S_d_value_set> class Config_manager
360 * f<S_d_value_set>(idx, dyn_else_st),
361 * ++idx,
362 * dyn_else_st = !dyn_else_st)
363 * )
364 * );
365 * ~~~
366 *
367 * `(..., expr)` expands to, ultimately, `(expr1, expr2, ..., expr3)`, where each `expr...i...` is `expr` with
368 * the i-th `S_d_value_set`. In our case `expr...i...` is itself a comma-expression with 3 sub-expressions.
369 * Comma-expression evaluates left-to-right, so at runtime the side effects of each
370 * `expr...i...` are occurring at runtime; in this case this includes incrementing `idx` and repeatedly flipping
371 * `dyn_else_st`.
372 *
373 * In the actual code, often we don't (just) call some helper `f<>()` but rather actually perform some steps in-line
374 * there instead -- similarly to how this example does a couple of simple expressions after calling `f<>()`.
375 * The good thing is the reader need not jump to another function (and all that maintenance overhead). The bad thing
376 * is the language constructions within an expression in C++ (not a functional language) are limited. Loops aren't
377 * possible; `if ()`s have to be replaced by `?:` ternaries or `&&` or `||` expressions; and so on.
378 * It's not always the most readable thing, if one isn't used to it, but the author feels it's doable to get used to it,
379 * and it's worth it for the brevity.
380 */

445 /**
446 * Invoke this after construction to load the permanent set of static config from config sources including
447 * a static config file; if you are also using dynamic config, see apply_static_and_dynamic() as a *potential*
448 * alternative. See also apply_static() overload with #allow_invalid_defaults_tag_t tag.
449 *
450 * After this runs and succeeds, assuming `commit == true`, you may use static_values() and/or
451 * all_static_values() to access the loaded values. After this runs and succeeds, but `commit == false`,
452 * you should call apply_static() again 1+ times, with the last such call having `commit == true` and thus
453 * canonicalizing each progressively-built candidate `Value_set`, so that it becomes available via static_values()
454 * and all_static_values().
455 *
456 * On failure returns `false`; else returns `true`. In the former case the canonical state remains unchanged,
457 * and any candidate built-up via preceding `commit == false` calls (if any) is discarded. The next
458 * `apply_*()` call begins a fresh update.
459 *
460 * Tip: On failure you may want to exit program with error; or you can continue knowing that
461 * static_values() will return a reference to default values (and all_static_values() will emit pointers to
462 * `Value_set`s with default values) according to `Value_set()` no-arg ctor (for each `Value_set`).
463 * WARNING(s) logged given failure.
464 *
465 * apply_static() will *not* be tolerant of unknown option names appearing in the config source. The reasoning
466 * (and we might change our minds over time): It should be possible to deliver the proper set of static config
467 * along with the binary that supports it. However apply_static_and_dynamic() is tolerant of unknown option names.
468 *
469 * You must supply exactly #S_N_S_VALUE_SETS `final_validator_func`s. The compiler will likely require you to
470 * explicitly specify the `struct` types as explicit template args; e.g:
471 *
472 * ~~~
473 * Config_manager<S_config1, D_config1, S_config2, D_config2, Null_value_set, D_config3> cfg_mgr(...);
474 * // Explicitly specify the static config struct types. Provide that number of final-validate functions.
475 * // Use null_final_validator_func() for the dummy Null_value_set in particular.
476 * cfg_mgr.apply_static<S_config1, S_config2, Null_value_set>
477 * (file_path, &s_decl_opts1, &s_decl_opts2, null_final_validator_func());
478 * ~~~
479 *
480 * For context w/r/t `commit` arg: Please read the section of the class doc header regarding multi-source
481 * updates. Corner case note: Formally: invoking `apply_Y()` immediately after `apply_X(commit = false) == true`,
482 * without reject_candidates() between them, where Y and X differ => `apply_Y()` shall log an INFO message
483 * and invoke reject_candidates() itself; then proceed. In this case `apply_Y()` is apply_static().
484 * Informally we discourage doing this; it is stylistically better to invoke reject_candidates()
485 * explicitly in that scenario which cancels an in-progress update which is unusual though conceivably useful.
486 *
487 * @note Each `final_validator_func()` can be made quite brief by using convenience macro
488 * FLOW_CFG_OPT_CHECK_ASSERT(). This will take care of most logging in most cases.
489 * @note For each Null_value_set: use `final_validator_func = null_final_validator_func()`.
490 * @note A validator function will not be run if there is a failure to parse any preceding `Value_set` (including a
491 * failure of its associated validator function).
492 * @note A validator function may rely on all_static_values_candidates() and static_values_candidate() providing the
493 * parsed (and validated) values candidate for any preceding static value set.
494 *
495 * @todo Add support for command line as a config source in addition to file(s), for static config in
496 * cfg::Config_manager and cfg::Static_config_manager.
497 *
498 * @tparam Value_set
499 * These must be `S_d_value_set` template param pack args 0, 2, ..., numbering
500 * #S_N_S_VALUE_SETS in order.
501 * @param static_cfg_path
502 * File to read.
503 * @param final_validator_func
504 * For each arg:
505 * If parsing and individual-option-validation succeed, the method shall return success if
506 * `final_validator_func(V)` returns Final_validator_outcome::S_ACCEPT or Final_validator_outcome::S_SKIP,
507 * where V is the parsed `Value_set`. (If `apply_static(commit = false) == true` call(s) preceded this
508 * one, V is the cumulative candidate from such call(s) so far, plus values parsed from
509 * `static_cfg_path` on top of that.)
510 * Informally: Please place individual-option validation
511 * into FLOW_CFG_OPTION_SET_DECLARE_OPTION() invocations; only use `final_validator_func()` for
512 * internal consistency checks (if any) and skip conditions (if any).
513 * @param commit
514 * `true` means that this call being successful (returning `true`) shall cause the promotion of
515 * each candidate `Value_set` built-up so far (via this call and all preceding successful calls with
516 * `commit == false`) to canonical state (accessed via static_values() or all_static_values()).
517 * `false` means that this call being successful shall merely create-and-set (if first such call)
518 * or incrementally update (if 2nd, 3rd, ... such call) each candidate `Value_set`.
519 * @return `true` if and only if successfully parsed config source and validated all settings including
520 * `final_validator_func() != S_FAIL` for *all* (static) config sets; and defaults were also all individually
521 * valid. However, if `true` but `commit == false`, then the canonical values
522 * (accessed via static_values() or all_static_values()) have not been updated.
523 * If `false` with `commit == false`, you should not call apply_static() for the planned
524 * subsequent config sources: this update has failed, and any built-up candidate `Value_set`s are
525 * discarded.
526 */
527 template<typename... Value_set>
528 bool apply_static(const fs::path& static_cfg_path,
529 const typename Final_validator_func<Value_set>::Type&... final_validator_func,
530 bool commit = true);

1218 /**
1219 * The static and dynamic value sets, in the same order as the #S_N_VALUE_SETS
1220 * `S_d_value_set` template args, the specific type for each slot being: `Option_set<S_d_value_set>`.
1221 * This is the main data store managed by Config_manager. Each static `S_d_value_set` (the even slots) is the
1222 * entirety of data having to do with that piece of static config. Each dynamic `S_d_value_set` (the odd slots) is
1223 * the *canonical* dynamic config; and a pointer to a *copy* thereof is also maintained in #m_d_value_sets.
1224 *
1225 * The slots, in order, are hence:
1226 * - Index [0]: static slot 0,
1227 * - Index [1]: dynamic slot 0,
1228 * - Index [2]: static slot 1,
1229 * - Index [3]: dynamic slot 1,
1230 * - ...
1231 * - Index [`.size() - 2`]: static slot `S_N_S_VALUE_SETS - 1`.
1232 * - Index [`.size() - 1`]: dynamic slot `S_N_D_VALUE_SETS - 1`.
1233 *
1234 * The actual type stored is `shared_ptr<void>`, with each slot's held pointer actually being an
1235 * `Option_set<S_d_value_set>*`.
1236 * opt_set() obtains the actual `Option_set<>` at a given slot, performing the required casting.
1237 *
1238 * ### Rationale ###
1239 * Why use an array of `shared_ptr<void>`, which requires a `static_cast` anytime we need to actually
1240 * access a config set? The more compile-time/meta-programm-y approach I (ygoldfel) could conceive of was to use
1241 * `tuple<S_d_value_set...>`. That would be quite elegant and lightning-fast (though compilation might be very slow).
1242 * However then, if I understand correctly, that would prevent any kind of runtime iteration through them.
1243 * This might be absolutely fine, if every slot were treated identically, but in the current setup it's
1244 * static, dynamic, static, dynamic, .... Therefore, by moving some processing to runtime instead of compile-time
1245 * we gain flexibility. (Do note my disclaimer in Implementation section in class doc header -- indicating this is
1246 * my first foray into advanced meta-programming like this. So perhaps I lack imagination/experience.)
1247 *
1248 * So then `shared_ptr<void>` actually stores an `S_d_value_set*` -- `S_d_value_set` being a different type for
1249 * each slot in the array. This uses the type erasure pattern of standard C++ smart pointers, wherein because
1250 * `static_cast<void*>(T*)` is allowed for any type `T`, `shared_ptr<void>` deleter will properly `delete` the `T*`,
1251 * as long as the `shared_ptr` ctor is passed a `T*` raw pointer. Do note this is type-safe only if our code
1252 * knows to cast to the proper `T*` when the pointer is accessed.
1253 *
1254 * Why `shared_ptr` -- not `unique_ptr` (which is nominally faster in that there's no ref-counting)? Answer:
1255 * `unique_ptr` is apparently deliberately simple and lacks aliasing and type-erasure constructors.
1256 * `unique_ptr<T>` can only delete a `T` with the default deleter; and `T` being `void` therefore won't compile.
1257 * It is probably possible to hack it with a custom deleter -- but this seemed like overkill, as these handles
1258 * are never used except inside an `array`, so reference counting shouldn't even happen and thus cost any perf
1259 * cycles outside of `*this` ctor and dtor.
1260 *
1261 * An alternative, which is somewhat more type-safe in that it would have resulted in an RTTI exception if the
1262 * wrong type-cast were used on the stored value, was `any`. However `any` is slower, involving `virtual` lookups
1263 * (at least in a naive implementation of `any`).
1264 */
1265 boost::array<Void_ptr, S_N_VALUE_SETS> m_s_d_opt_sets;
1266
1267 /**
1268 * The baseline dynamic value sets, in the same order as #S_N_D_VALUE_SETS *dynamic*
1269 * `S_d_value_set` template args, the specific type for each slot being `S_d_value_set`.
1270 *
1271 * The actual type stored is `shared_ptr<void>`, with each slot's held pointer actually being `S_d_value_set*`.
1272 * The same type erasure technique used in #m_s_d_opt_sets is used here.
1273 * d_baseline_value_set() obtains the actual `S_d_value_set` at a given slot, performing the required casting.
1274 *
1275 * A null pointer is stored if the corresponding Option_set in #m_s_d_opt_sets is `null()` (a placeholder with 0
1276 * settings).
1277 *
1278 * ### Rationale / explanation ###
1279 * Suppose a dynamic `Value_set` is to be parsed from file F potentially repeatedly, as dynamic updates come in.
1280 * Then its Option_set from `m_s_d_opt_sets[]` would `parse_config_file(F)` each time. However, if that is all
1281 * we did, then the updates would stack on top of each other incrementally: if one looks at F's contents at a given
1282 * time, one cannot tell what the `Value_set` in memory would contain upon parsing it; it would depend on any
1283 * preceding updates. That's bad: the resulting memory `Value_set` must be consistent, regardless of what
1284 * F contained in preceding updates. Therefore, a certain *baseline* state for that `Value_set` must be loaded
1285 * into the Option_set just before the Option_set::parse_config_file() call in each update.
1286 *
1287 * This stores each dynamic slot's baseline `Value_set` state to load as just described.
1288 *
1289 * After construction this payload is just the default-cted `Value_set()`. If the user chooses to execute
1290 * a one-time apply_static_and_dynamic(), then that payload is replaced by the state after having parsed that
1291 * baseline state. Note, for context, that apply_static_and_dynamic() would be presumably loading *not*
1292 * from file F (which can change repeatedly, as dynamic updates come in) but some other file B, typically storing
1293 * both static value sets' contents as well as the baseline dynamic value sets'.
1294 */
1295 boost::array<Void_ptr, S_N_D_VALUE_SETS> m_d_baseline_value_sets;

1297 /**
1298 * The dynamic config ref-counted handles returned by all_dynamic_values(). Each one is a ref-counted pointer
1299 * to an immutable *copy* of the canonical `S_d_value_set` in that *dynamic* slot of #m_s_d_opt_sets
1300 * (namely `m_s_d_opt_sets[2 * d_idx + 1]`, where `d_idx` is in [0, 1, ..., #S_N_D_VALUE_SETS)).
1301 *
1302 * The actual type stored is `shared_ptr<void>`, with each slot's held pointer actually being
1303 * `S_d_value_set::Const_ptr*`. The same type erasure technique used in #m_s_d_opt_sets is used here.
1304 *
1305 * Protected by #m_d_value_sets_mutex.
1306 *
1307 * apply_dynamic() and apply_static_and_dynamic() set these; all_dynamic_values() and dynamic_values() get them.
1308 *
1309 * ### Rationale ###
1310 * Why is this necessary? Why not just expose a pointer to immutable `S_d_value_set` directly inside
1311 * `m_s_d_opt_sets[d_idx]` as we do (via static_values() and all_static_values()) for the static config? Answer:
1312 * Because with dynamic config it is important that the user be able to retain (if even for a short time, perhaps to
1313 * complete some operation -- *during* which `*this` might process a dynamic update via apply_dynamic()) an immutable
1314 * `S_d_value_set`. Since by definition that's not possible with dynamic config updates, we must create a copy. At
1315 * that point it becomes natural to wrap it in a `shared_ptr`, so that only *one* copy is necessary in the entire
1316 * application -- and it'll disappear once (1) the canonical config has been updated, replacing `m_d_value_sets[]`,
1317 * and (2) all user code has dropped the remaining refs to it.
1318 *
1319 * The ref-counted-pointer-to-dynamic-config is a common pattern. The only thing we add that might not be obvious
1320 * is having the canonical copy be separate from the one we expose via the pointer publicly. The latter is
1321 * the natural consequence of the fact that the `Value_set` managed by `Option_set<Value_set>` lives inside
1322 * the `Option_set` and is not wrapped by any `shared_ptr`. However Option_set::mutable_values_copy() does provide
1323 * easy access to a `shared_ptr`-wrapped copy thereof... so we use it. The perf cost is that of a value-set
1324 * copy at apply_dynamic() time which is negligible in the grand scheme of things.
1325 */

1347 /**
1348 * In short, truthy if and only if a `commit == false` update is currently in progress,
1349 * meaning there's an uncommitted *candidate* `Value_set` at the moment. More precisely:
1350 * - If the last-invoked `apply_*()` method (1) returned `false`, or (2) returned `true` but with arg
1351 * `commit == true`, or (3) has never been called, or (4) was followed at any point
1352 * by reject_candidates(), then this equals Update_type::S_NONE. I.e., no multi-source update is
1353 * currently in progress. Otherwise:
1354 * - This indicates which `apply_*(commit = true) == true` method (multi-source update type) it was.
1355 * To the extent the exact type is irrelevant, its being not NONE indicates a multi-source update
1356 * is in progress.
1357 *
1358 * Inside an `apply_*()` method, this is updated to its new value after parsing into #m_s_d_opt_sets. Thus
1359 * during the bulk of such a method's processing we can easily determine whether this is step 1 of a
1360 * (potentially multi-source) update -- rather than step 2+ -- as this will equal NONE in the former
1361 * case only.
1362 *

1763template<typename... S_d_value_set>
1764template<typename Value_set>
1765bool Config_manager<S_d_value_set...>::apply_impl
1766 (Option_set<Value_set>* opt_set,
1767 const Value_set* baseline_value_set_or_null,
1768 const fs::path& cfg_path,
1769 const boost::unordered_set<std::string>& all_opt_names_or_empty,
1770 const typename Final_validator_func<Value_set>::Type& final_validator_func)
1771{
1772 assert(opt_set);
1773 auto& opts = *opt_set;
1774
1775 if (opts.null())
1776 {
1777 return true; // By the way notice, as advertised, final_validator_func is ignored.
1778 }
1779 // else
1780
1781 /* For some sets (namely dynamic ones, at most) we need to pre-apply the full set of baseline values.
1782 * This is explained in m_d_baseline_value_sets doc header; but basically it's so that successive dynamic updates
1783 * always apply the change in the file on top of a consistent state. Otherwise it's impossible to know, just by
1784 * looking at the dynamic config file, what the resulting state will be in memory -- incremental is bad in this
1785 * context. */
1786 if (baseline_value_set_or_null)
1787 {
1788 opts.parse_direct_values(*baseline_value_set_or_null);
1789 }
1790
1791 /* We allow final_validator_func()==SKIP, which means the validator decided the candidate Value_set is not
1792 * erroneous but has something there that means this particular apply_impl() shall have no effect on the candidate
1793 * after all. To make this work we must memorize the pre-parse state; as Option_set doesn't know about
1794 * final-validation or skipping and will have updated .values_candidate() (otherwise) irreversibly. We, however,
1795 * can simply undo it via .parse_direct_values() or .reject_candidate().
1796 *
1797 * Subtlety: If, in this possibly-multi-source update, this is the 1st source, then opt_set->values_candidate() may be
1798 * null (*opt_set is not in PARSING state but CANONICAL still). One option is to do what we do: save the canonical
1799 * value-set instead; then if rewinding below, it'll stay in PARSING state but have the unchanged canonical values.
1800 * Another option is to, instead, save nothing and when rewinding below in that case
1801 * reject_candidate() and thus bring *opt_set to CANONICAL state again. That may seem a bit more complex but more
1802 * elegant, but it leads to a subtle bug: If the possibly-multi-source update's final-validators *all* return SKIP --
1803 * which is arguably unusual but certainly allowed -- then apply_*() near its end will happily attempt to
1804 * opt_set->canonicalize_candidate() -- which will fail (possibly assert-trip) due to being invoked in
1805 * already-CANONICAL state. So our choices are to go with the just-save-values() route, or to -- elsewhere
1806 * in apply_*() themselves -- guard ->canonicalize_candidate() with a check for the everything-SKIPped situation.
1807 * Both are okay, but the former approach is simpler, avoiding at least a couple of if-branches in a couple of
1808 * places. It involves potentially 2 extra copy-ops of a Value_set; but so what? In the grand scheme the perf
1809 * impact is negligible. */
1810 const auto candidate_or_null = opt_set->values_candidate();
1811 const auto pre_parse_candidate = candidate_or_null ? *candidate_or_null : opt_set->values();
1812
1813 /* We allow unknown (to this Option_set<Value_set> `opts`) options in the file during the initial parse -- but
1814 * only the ones that are themselves options but in other Option_sets we're also parsing (since non-hard-coded options
1815 * are not being parsed in this file, no other options should be present -- forward-compatibility, etc., not a
1816 * factor with a hard-coded file). Hence simply pass in all option names (including this one's, but they'll
1817 * just be parsed as belonging to this actual set and hence not be relevant to the unknown-option-name check).
1818 *
1819 * If it's not an initial parse then all_opt_names_or_empty will be empty, meaning simply allow all unknown options;
1820 * due to forward/backward-compatibility we just allow it and log such things but ignore them otherwise. */
1821 bool ok;
1822 opts.parse_config_file(cfg_path, true, &ok, all_opt_names_or_empty);
1823
1824 /* parse_config_file() specifically promises not to individually-validate values not in the actual file.
1825 * So before returning success (likely causing to finalize values via opts.canonicalize_candidate()) ensure
1826 * any invalid defaults don't "fall through." This is unnecessary if defaults were scanned individually
1827 * due to allow_invalid_defaults==false; but in that case it is harmless (as of this writing won't even log). */
1828 if (ok)
1829 {
1830 opts.validate_values_candidate(&ok);
1831 if (!ok) // It would have logged the exact problem, but we still want to clearly say what happened, high-level:
1832 {
1833 FLOW_LOG_WARNING("Config_manager [" << *this << "]: "
1834 "Request to apply config file [" << cfg_path << "] for a particular value set: "
1835 "Individual values present in file were OK, but at least 1 default value (see above) "
1836 "was invalid. Either correct the default or ensure it is properly overridden in file.");
1837 }
1838 } // if (ok) (but it may have become false inside)
