86/**
87 * Short-hand for boost.asio event service, the central class of boost.asio.
88 *
89 * ### Naming ###
90 * The reasons for the rename -- as opposed to simply calling it `Io_service` (historically it was indeed so named) --
91 * are as follows. To start, a later version of Boost (to which we eventually moved) realigned
92 * boost.asio's API to match the feedback boost.asio received when submitted as a candidate for inclusion into
93 * the official STL (sometime after C++17); in this reorg they renamed it from `io_service` to `io_context`.
94 * That's not the actual reason but more of a symptom of similar reasoning to the following:
95 * - "Service" sounds like something that has its own thread(s) perhaps, which `io_service` actually doesn't;
96 * it *can* (and typically is) used to take over at least one thread, but that's not a part of its "essence," so
97 * to speak. I've noticed that when boost.asio neophytes see "service" they intuitively draw incorrect assumptions
98 * about what it does, and then one must repeatedly dash those assumptions.
99 * - Furthermore (though apparently boost.asio guys didn't agree or didn't agree strongly enough) "I/O" implies
100 * an `io_service` is all about working with various input/output (especially networking). While it's true
101 * that roughly 50% of the utility of boost.asio is its portable sockets/networking API, the other 50% is about
102 * its ability to flexibly execute user tasks in various threads; and, indeed, `io_service` *itself* is more about
103 * the latter 50% rather than the former. Its two most basic and arguably oft-used features are simply
104 * `io_service::post()` (which executes the given task ASAP); and the `basic_waitable_timer` set of classes
105 * (which execute task at some specified point in the future). Note these have nothing to do with networking or
106 * I/O of any kind. The I/O comes from the "I/O object" classes/etc., such as `tcp::socket`, but those
107 * are *not* `io_service`; they merely work together with it to execute the user-specified success/failure handlers.
108 *
109 * So, neither "I/O" nor "service" is accurate. To fix both, then, we used this rename reasoning:
110 * - "I/O" can be replaced by the notion of it executing "tasks." boost.asio doesn't use or define the "task" term,
111 * sticking to "handler" or "functor" or "function," but we feel it's a reasonable term in this context.
112 * (Function, etc., doesn't cover all possibilities. Handler is OK, but when it's not in response to any event --
113 * as with vanilla `post()` -- what is it handling?)
114 * - "Service" is trickier to replace. It's a battle between "too generic" and "too specific and therefore long to
115 * type." Plus there's the usual danger of accidentally overloading a term already used elsewhere nearby.
116 * Some ideas considered were: manager, engine, execution engine, queue, queues, context (which boost.asio chose),
117 * and various abbreviations thereof. Ultimately it came down to context vs. engine, and I chose engine because
118 * context is just a little vague, or it suggests an arbitrary grouping of things -- whereas "engine" actually
119 * implies action: want to express this thing provides logical/algorithmic value as opposed to just bundling
120 * stuff together as "context" objects often do (e.g., "SSL context" in openssl). Indeed `io_service` does provide
121 * immense algorithmic value.
122 * - Manager: generic but not bad; clashes with "Windows task manager" and such.
123 * - Execution engine: good but long.
124 * - Queue: early candidate, but at the very least it's not a *single* queue; if there are multiple threads
125 * `io_service::run()`ning then to the extent there are any queues there might be multiple.
126 * - Queues, qs: better... but having a thing called `Task_qs` -- of which there could be multiple (plural) --
127 * was clumsy. (An early proof-of-concept used `Task_qs`; people were not fans.)
128 */
129using Task_engine = boost::asio::io_service;

134/**
135 * boost.asio timer. Can schedule a function to get called within a set amount of time or at
136 * a certain specific time relative to some Epoch.
137 *
138 * @see schedule_task_from_now() (and friends) for a simpler task-scheduling API (which internally uses this
139 * `Timer). The notes below still apply however. Please read them even if you won't use `Timer` directly.
140 *
141 * Important: While one can schedule events using very precise units using #Fine_duration, the
142 * timer may not actually have that resolution. That is, I may schedule something to happen in 1
143 * millisecond and then measure the time passed, using a high-res clock like #Fine_clock, and discover that
144 * the wait was actually 15 milliseconds. This is not a problem for most timers. (In Flow itself, the most advanced
145 * use of timers by far is in flow::net_flow. Timers used in flow::net_flow -- delayed ACK timer, Drop Timer, simulated
146 * latency timer -- typically are for `n x 50` milliseconds time periods or coarser.) However that may not be the
147 * case for all timers. (In particular, flow::net_flow packet pacing may use finer time periods.)
148 *
149 * @note The flow::net_flow references in the present doc header speak of internal implementation details -- not
150 * of interest to the `net_flow` *user*. I leave them in this public doc header for practical purposes, as
151 * examples needed to discuss some Inside-Baseball topics involved.
152 *
153 * So I looked into what boost.asio provides. `deadline_timer` uses the system clock
154 * (`universal_time()`) as the time reference, while `basic_waitable_timer<Fine_clock>` uses the
155 * high-resolution clock (see #Fine_clock). I tested both (using `wait()`s of various lengths
156 * and using #Fine_clock to measure duration) and observed the following resolutions on certain OS
157 * (not listing the hardware):
158 *
159 * - Linux (2.6.x kernel): sub-2 millisecond resolution, with some variance.
160 * (Note: This Linux's `glibc` is too old to provide the `timerfd` API, therefore Boost falls back to
161 * using `epoll_wait()` directly; on a newer Linux this may get better results.)
162 * - Windows 7: 15 millisecond resolution, with little variance.
163 * - Mac OS X: untested. @todo Test Mac OS X timer fidelity.
164 *
165 * These results were observed for BOTH `deadline_timer` and `basic_waitable_timer<Fine_clock>`
166 * in Boost 1.50. Thus, there was no resolution advantage to using the latter -- only an interface advantage.
167 * Conclusion: we'd be well-advised not to rely on anything much smaller than 20 milliseconds when
168 * scheduling timer events. One technique might be, given any time T < 20 ms, to assume T = 0
169 * (i.e., execute immediately). This may or may not be appropriate depending on the situation.
170 *
171 * However, using `basic_waitable_timer<Fine_clock>` may get an additional nice property: #Fine_clock
172 * always goes forward and cannot be affected by NTP or user changes. `deadline_timer` may or may not
173 * be affected by such tomfoolery (untested, but the potential is clearly there). Therefore we do the best
174 * we can by using the #Fine_clock-based timer.
175 *
176 * @todo Upgrade to newer Boost and keep testing timer resolutions on the above and other OS versions.
177 * Update: As of this writing we are on Boost 1.74 now. Needless to say, it is time to reassess the above, as it
178 * has been years since 1.50 -- and we had been on 1.63 and 1.66 for long period of time, let alone 1.74.
179 * Update: No changes with Boost 1.75, the newest version as of this writing.
180 *
181 * @note One annoying, though fully documented, "feature" of this and all boost.asio timers is that `cancel()`ing it
182 * does not always cause the handler to fire with `operation_aborted` error code. Sometimes it was already "just
183 * about" to fire at `cancel()` time, so the handler will run with no error code regardless of the `cancel()`.
184 * Personally I have never once found this behavior useful, and there is much code currently written to work
185 * around this "feature." Furthermore, even a "successful" cancel still runs the handler but with `operation_aborted`;
186 * 99.999% of the time one would want to just not run it at all. Finally, in many cases, even having to create
187 * a timer object (and then the multiple steps required to actually schedule a thing) feels like more boiler-plate
188 * than should be necessary. To avoid all of these usability issues, see schedule_task_from_now() (and similar),
189 * which is a facility that wraps `Timer` for a boiler-plate-free experience sufficient in the majority of practical
190 * situations. `Timer` is to be used directly when that simpler facility is insufficient.
191 *
192 * @note `b_w_t<Fine_clock>` happens to equal `boost::asio::high_resolution_timer`. We chose to alias
193 * in terms of #Fine_clock merely to ensure that #Fine_duration is one-to-one compatible with it, as it
194 * is defined in terms of #Fine_clock also.
195 */
196using Timer = boost::asio::basic_waitable_timer<Fine_clock>;

214/**
215 * Short-hand for non-reentrant, shared-or-exclusive mutex.
216 * When locking one of these, choose one of:
217 * `Lock_guard<Mutex_shared_non_recursive>`, `Shared_lock_guard<Mutex_shared_non_recursive>`.
218 * The level of locking acquired (shared vs. exclusive) depends on which you chose and is thus highly significant.
219 *
220 * @todo Consider changing util::Mutex_shared_non_recursive from `boost::shared_mutex` to `std::shared_mutex`, as
221 * the former has a long-standing unresolved ticket about its impl being slow and/or outdated (as of Boost-1.80).
222 * However see also the note on util::Mutex_shared_non_recursive that explains why it might be best to avoid
223 * this type of mutex altogether in the first place (in most cases).
224 *
225 * ### When to use versus #Mutex_non_recursive ###
226 * Experts suggest fairly strongly that one should be very wary about using a shared mutex, over a simple
227 * non-recursive exclusive mutex, in almost any practical application. The reason there's a trade-off at all is
228 * that a #Mutex_non_recursive is extremely fast to lock and unlock; the entire perf cost is in waiting for
229 * another thread to unlock first; so without contention it's almost *free*. In contrast apparently any real
230 * impl of #Mutex_shared_non_recursive is much slower to lock/unlock. The trade-off is against allowing
231 * reads to proceed in parallel; but experts say this is "worth it" only when these read critical sections
232 * are lengthy and very frequently invoked. So, for example, if the lock/unlock is around an `unordered_map`
233 * lookup with a quick hashing function, it would be quite difficult to induce enough lock contention
234 * to make a shared mutex better than an exclusive one. (Note this is assuming even no perf issue with
235 * `boost::shared_mutex` specifically.)
236 */
237using Mutex_shared_non_recursive = boost::shared_mutex;

349/**
350 * Get the current POSIX (Unix) time as a duration from the Epoch time point.
351 *
352 * This is the amount of time -- according to the user-settable system clock time -- to have passed since the
353 * POSIX (Unix) Epoch -- January 1st, 1970, 00:00:00 UTC -- not counting leap seconds to have been inserted or deleted
354 * between then and now.
355 *
356 * The `boost::chrono` duration type is chosen so as to support the entire supported resolution of the
357 * OS-exposed system clock (but probably no more than that).
358 *
359 * ### Known use cases, alternatives ###
360 * - Logging of time stamps. Output the raw value; or use boost.locale to output `ceil<seconds>()` in the desired
361 * human-friendly form, splicing in the left-over microseconds where desired (boost.locale lacks formatters for
362 * sub-second-resolution time points). However see below for a typically-superior alternative.
363 * - By subtracting return values of this at various points in time from each other, as a crude timing mechanism.
364 * (Various considerations make it just that -- crude -- and best replaced by flow::Fine_clock and the like.
365 * Moreover see flow::perf::Checkpointing_timer.)
366 * - It's a decent quick-and-dirty random seed.
367 *
368 * ### Update/subtleties re. time stamp output ###
369 * Using this for time stamp output is no longer needed or convenient, as a much nicer way presents itself
370 * when combined with boost.chrono I/O-v2 `time_point`-outputting `ostream<<` overload. Just grab `time_point`
371 * `boost::chrono::system_clock::now()` (or another `system_clock`-originated value); its default-formatted
372 * `ostream<<` output will include date, time with microsecond+ precision, and time-zone specifier (one can choose
373 * UTC or local time).
374 *
375 * However, as of this writing, it is not possible to directly obtain *just* the microsecond+
376 * part of this, in isolation, according boost.chrono docs. (One could hack it by taking a substring.)
377 * Quote: "Unfortunately there are no formatting/parsing sequences which indicate fractional seconds." From:
378 * https://www.boost.org/doc/libs/1_76_0/doc/html/chrono/users_guide.html#chrono.users_guide.tutorial.i_o.system_clock_time_point_io)
379 *
380 * @return A duration representing how much time has passed since the Epoch reference point
381 * (could be negative if before it).
382 */
383boost::chrono::microseconds time_since_posix_epoch();

385/**
386 * Writes a multi- or single-line string representation of the provided binary data to an output stream, complete with
387 * a printable and hex versions of each byte.
388 * - Single-line mode is chosen by setting `bytes_per_line` to special value `-1`. In this mode, the output
389 * consists of `indentation`, followed by a pretty-printed hex dump of every byte in `data` (no matter how many),
390 * followed by a pretty-printed ASCII-printable (or '.' if not so printable) character for each byte; that's it.
391 * No newline is added, and `indentation` is included only the one time. It is recommended to therefore print
392 * "in-line" inside log messages (etc.) without surrounding newlines, etc., when using this mode.
393 * - Multi-line mode is chosen by setting `bytes_per_line` to a positive value; or 0 which auto-selects a decent
394 * default positive value. Then, each line represents up to that many bytes, which are pretty-printed similarly
395 * to single-line mode, in order of appearance in `data` (with the last such line representing the last few bytes,
396 * formatting nicely including accounting for potentially fewer than the desired # of bytes per line).
397 * Every line starts with `indentation` and ends with a newline -- including the last line.
398 * Therefore, it is recommended to precede this call with an output of a newline and to avoid doing so
399 * after that call (unless blank line is desired).
400 *
401 * Example with a single contiguous memory area, multi-line mode:
402 *
403 * ~~~
404 * array<uint8_t, 256 * 256> bytes(...);
405 *
406 * // Output to cout.
407 * cout
408 * << "Buffer contents: [\n";
409 * buffers_to_ostream(cout,
410 * boost::asio::buffer(bytes), // Turn a single memory array into a buffer sequence.
411 * " ");
412 * << "]."; // This will be on its own line at the end.
413 * ~~~
414 *
415 * See also buffers_dump_string() which returns a string and can thus be more easily used directly inside
416 * FLOW_LOG_DATA() and similar log macros.
417 *
418 * ### Performance ###
419 * This thing is slow... it's not trying to be fast and can't be all that fast anyway. As usual, though, if used
420 * in FLOW_LOG_DATA() (etc.) its slowness will only come into play if the log filter passes which (esp. for
421 * log::Severity::S_DATA) it usually won't.
422 *
423 * That said buffers_dump_string() is even slower, because it'll use an intermediate `ostream` independent of whatever
424 * `ostream` you may or may not be placing the resulting string. However it's easier to use with `FLOW_LOG_...()`,
425 * and since in that case perf is typically not an issue, it makes sense to use the easier thing typically.
426 *
427 * However, if you do want to avoid that extra copy and need to *also* use buffers_to_ostream() directly inside
428 * `FLOW_LOG_...()` then the following technique isn't too wordy and works:
429 *
430 * ~~~
431 * const array<uint8_t, 256 * 256> bytes(...);
432 *
433 * // Log with a flow::log macro.
434 * const flow::Function<ostream& (ostream&)> os_manip = [&](ostream& os) -> ostream&
435 * {
436 * return buffers_to_ostream(os, boost::asio::buffer(bytes), " ");
437 * };
438 * FLOW_LOG_INFO("Buffer contents: [\n" << os_manip << "].");
439 *
440 * // Above is probably more performant than:
441 * FLOW_LOG_INFO("Buffer contents: [\n"
442 * << buffers_dump_string(boost::asio::buffer(bytes), " ") // Intermediate+copy, slow....
443 * << "].");
444 * // flow::util::ostream_op_string() is a bit of an improvement but still. :-)
445 * ~~~
446 *
447 * ### Rationale ###
448 * The reason it returns `os` and takes a reference-to-mutable instead of the customary (for this project's
449 * style, to indicate modification potential at call sites) pointer-to-mutable is in order to be `bind()`able
450 * in such a way as to make an `ostream` manipulator. In the example above we use a lambda instead of `bind()`
451 * however.
452 *
453 * @tparam Const_buffer_sequence
454 * Type that models the boost.asio `ConstBufferSequence` concept (see Boost docs) which represents
455 * 1 or more scattered buffers in memory (only 1 buffer in a sequence is common).
456 * In particular `boost::asio::const_buffer` works when dumping a single buffer.
457 * @param os
458 * The output stream to which to write.
459 * @param data
460 * The data to write, given as an immutable sequence of buffers each of which is essentially a pointer
461 * and length. (Reminder: it is trivial to make such an object from a single buffer as well;
462 * for example given `array<uint8_t, 256> data` (an array of 256 bytes), you can just pass
463 * `boost::asio::buffer(data)`, as the latter returns an object whose type satisfies requirements of
464 * `Const_buffer_sequence` despite the original memory area being nothing more than a single
465 * byte array.)
466 * @param indentation
467 * The indentation to use at the start of every line of output.
468 * @param bytes_per_line
469 * If 0, act as-if a certain default positive value were passed; and then: If `-1`, single-line mode is invoked.

978 * - The invocation is written as if M is a free function in the C language, and it is being "called"
979 * as the *entirety* of the statement containing that "call." That is, it must look like this:
980 *
981 * ~~~
982 * M(...arg 1 value..., ...arg 2 value..., ...more...); // (Or however many args M actually allows.)
983 * ~~~
984 *
985 * ### Known use case 1: Macro definition is 2+ statements ###
986 * Suppose your macro's value is a series of two or more statements executing in series without braces `{}`
987 * around the whole thing. Then, in some contexts, invoking that macro as intended might result in
988 * unexpected runtime misbehavior that the compiler wouldn't detect. A simple example with two
989 * expressions as individual statements:
990 *
991 * ~~~
992 * #define DOUBLE_AND_PRINT(arg) \
993 * arg *= 2; // Statement 1. \
994 * cout << arg // Statement 2. Note the lacking trailing semicolon (invoker must supply it after "call"). \
995 * DOUBLE_AND_PRINT(x); // No problem. Both statements execute.
996 * if (some_flag)
997 * DOUBLE_AND_PRINT(x); // UNINTENDED BEHAVIOR! Statement 2 executes regardless of some_flag!
998 * ~~~
999 *
1000 * Granted, if the invoker used braces appropriately around the `if` statement body, there'd be no issue, but one
1001 * cannot always rely on invoker's good style, especially if the macro is part of a public library API.
1002 * Solution:
1003 *
1004 * ~~~
1005 * #define DOUBLE_AND_PRINT(arg) \
1006 * FLOW_UTIL_SEMICOLON_SAFE \
1007 * ( \
1008 * arg *= 2; // Statement 1. \
1009 * cout << arg; // Statement 2. Note necessity for trailing semicolon. \
1010 * )
1011 * DOUBLE_AND_PRINT(x); // No problem. Both statements execute.
1012 * if (some_flag)
1013 * DOUBLE_AND_PRINT(x); // No problem. Both statements execute; or neither one does.
1014 * ~~~
1015 *
1016 * One might ask: Why not simply surround the 2+ statements with braces `{}` to eliminate the problem more simply?
1017 * Answer: This actually would turn the use case into the following other problematic use case.
1018 *
1019 * ### Known use case 2: Required semicolon `;` trailing macro invocation leads to syntax error ###
1020 * Recall that the premise *requires* invoker to call your macro M like `M();`, meaning they will *always* put
1021 * a semicolon after the invocation. Now suppose that doing so will turn one statement (which is M's definition)
1022 * into two. Another way of putting it: Suppose M's body is already a complete statement, even without the required
1023 * trailing `;` (and there is no way to make it otherwise; usually [always?] because it ends with a `}`).
1024 * Then in some contexts it'll cause a syntax error. Example:
1025 *
1026 * ~~~
1027 * #define CHECK_AND_LOG(msg) \
1028 * if (filter()) \
1029 * { \
1030 * log(msg); \
1031 * } // Semicolon will always be added right here.
1032 * CHECK_AND_LOG("Hello."); // No problem. The empty ; statement introduced is harmless.
1033 * if (terminating)
1034 * CHECK_AND_LOG(msg); // SYNTAX ERROR! The empty ; will make the `else` illegal.
1035 * else
1036 * exit();
1037 * ~~~
1038 *
1039 * Removing the `;` from the invocation is no solution. Firstly it'll confuse many editors that will think the
1040 * "function call" (really macro invocation) is missing the usual semicolon. This might cause incorrect
1041 * auto-indentation in the file and who knows what else. Code analyzer tools may also be affected.
1042 * Even ignoring issues where the code is fed into a non-compiler, there are possible flow control problems.
1043 * In this example it'll attach the `else` in the invoking code to the inner `if` (from the macro)
1044 * instead of the outer `if` (in the invoking code) as intended. Wrapping the body of `CHECK_AND_LOG()` in
1045 * the proposed wrapper will resolve all stated problems. (Solution code omitted for brevity and obviousness
1046 * at this point.)
1047 *
1048 * @note Adding this wrapper would slightly adversely affect performance in the complete absence of compiler
1049 * optimization, but even the most basic optimizer (one capable of eliminating `if (false)` and similar)
1050 * should entirely eliminate any such performance degradation.
1051 *
1052 * @param ARG_func_macro_definition
1053 * The intended value of a functional macro, such that the "function" it approximates would have
1054 * return type `void`. Behavior is undefined if the value of this parameter ends in a semicolon, or
1055 * if the value contains a statement that includes another statement within it that is not a block
1056 * (`{ ... }`). (The rules in the last [English] statement may or may not be strictly necessary,
1057 * but requiring these allows me to generalize/reason more straightforwardly. In any case, one rule is
1058 * assumed good style anyway, while the other is simply practical in this context.)
1059 * @return Code to be used as the entire definition of a macro whose definition would have been
1060 * `ARG_func_macro_definition` if not for the resulting compile error or unexpected flow control
1061 * effects detailed above.
1062 */
1063#define FLOW_UTIL_SEMICOLON_SAFE(ARG_func_macro_definition) \
1064 do \
1065 { \
1066 ARG_func_macro_definition \
1067 } \
1068 while (false)
