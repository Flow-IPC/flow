33/**
34 * An implementation of Logger that logs messages to a given file-system path but never blocks any logging thread
35 * for file I/O; suitable for heavy-duty file logging. Protects against garbling due to simultaneous logging from
36 * multiple threads.
37 *
38 * For file logging, the two out-of-the-box `Logger`s currently suitable are this Async_file_logger and
39 * Simple_ostream_logger. Vaguely speaking, Simple_ostream_logger is suitable for console (`cout`, `cerr`) output;
40 * and, in a pinch outside of a heavy-duty production/server environment, for file (`ofstream`) output. For heavy-duty
41 * file logging one should use this Async_file_logger. The primary reason is performance; this is discussed
42 * in the Logger class doc header; note Async_file_logger logs asynchronously.
43 * A secondary reason is additional file-logging-specific utilities -- such as
44 * rotation -- are now or in the future going to be in Async_file_logger, as its purpose is heavy-duty file logging
45 * specifically.
46 *
47 * ### Thread safety ###
48 * As noted above, simultaneous logging from multiple threads is safe from output corruption, in that
49 * simultaneous do_log() calls for the same Logger targeting the same stream will log serially to each other.
50 * However, if some other code or process writes to the same file, then all bets are off -- so don't.
51 *
52 * See thread safety notes and to-dos regarding #m_config in Simple_ostream_logger doc header. These apply here also.
53 *
54 * There are no other mutable data (state), so that's that.
55 *
56 * @todo Lacking feature: Compress-as-you-log in Async_file_logger. So, optionally, when characters are actually
57 * written out to file-system, gzip/zip/whatever them instead of writing plain text. (This is possible at least
58 * for gzip.) Background: It is common-place to compress a log file after it has been rotated (e.g., around rotation
59 * time: F.log.1.gz -> F.log.2.gz, F.log -> F.log.1 -> F.log.1.gz). It is more space-efficient (at least), however,
60 * to write to F.log.gz directly already in compressed form; then rotation requires only renaming (e.g.:
61 * F.log.1.gz -> F.log.2.gz, F.log.gz [already gzipped from the start] -> F.log.1.gz).
62 */

70 /**
71 * Constructs logger to subsequently log to the given file-system path. It will append.
72 *
73 * @todo Consider adding Async_file_logger constructor option to overwrite the file instead of appending.
74 *
75 * @param config
76 * Controls behavior of this Logger. In particular, it affects should_log() logic (verbosity default and
77 * per-component) and output format (such as time stamp format). See thread safety notes in class doc header.
78 * This is saved in #m_config.
79 * @param log_path
80 * File-system path to which to write subsequently. Note that no writing occurs until the first do_log() call.
81 * @param backup_logger_ptr
82 * The Logger to use for `*this` to log *about* its logging operations to the actual intended file-system path;
83 * or null to not log such things anywhere. If you do not pass in null, but ensure
84 * `backup_logger_ptr->should_log()` lets through nothing more than `Sev::S_INFO` severity messages for
85 * `Flow_log_component::S_LOG`, then you can expect a reasonable amount of useful output that will not
86 * affect performance. Tip: null is a reasonable value. A Simple_ostream_logger logging to `cout` and `cerr`
87 * (or only `cout`) is also a good choice, arguably better than null.
88 * Lastly, setting verbosity to `INFO` for `*backup_logger_ptr` is typically a better choice than
89 * `TRACE` in practice.
90 * @param capture_rotate_signals_internally
91 * If and only if this is `true`, `*this` will detect SIGHUP (or your OS's version thereof);
92 * upon seeing such a signal, it will fire the equivalent of log_flush_and_reopen(), as needed for classic
93 * log rotation. (The idea is: If we are writing to path F, then your outside log rotation tool will rename
94 * F -> F.1 [and F.1 -> F.2, etc.]; even as we continue writing to the underlying file after it has been
95 * renamed; then the tool sends SIGHUP; we flush/close what is really F.1; reopen at the real path F
96 * again, which will create it anew post-rotation.) If `false` then you'd have to do it yourself if desired.
97 * If this is `true`, the user may register their own signal handler(s) (for any purpose whatsoever) using
98 * `boost::asio::signal_set`. However, behavior is undefined if the program registers signal handlers via any
99 * other API, such as `sigaction()` or `signal()`. If you need to set up such a non-`signal_set` signal
100 * handler, AND you require rotation behavior, then (1) set this option to `false`; (2) trap SIGHUP yourself;
101 * (3) in your handlers for the latter, simply call log_flush_and_reopen(). However, if typical, common-sense
102 * behavior is what you're after -- and either don't need additional signal handling or are OK with using
103 * `signal_set` for it -- then setting this to `true` is a good option.
104 */
105 explicit Async_file_logger(Logger* backup_logger_ptr,
106 Config* config, const fs::path& log_path,
107 bool capture_rotate_signals_internally);

145 /**
146 * Causes the log at the file-system path to be flushed/closed (if needed) and
147 * re-opened; this will happen as soon as possible but may occur asynchronously after this method exits, unless
148 * directed otherwise via `async` argument.
149 *
150 * ### Uses ###
151 * Flushing: `log_flush_and_reopen(false)` is a reasonable and safe way to flush anything buffered in memory to the
152 * file. Naturally, for performance, it should not be done frequently. For example this might be useful in the
153 * event of an abnormal termination (`abort()`, etc.), in the signal handler before exiting program.
154 *
155 * Rotation: `log_flush_and_reopen(true)` is useful for rotation purposes; however, you need not do this manually if
156 * you decided to (properly) use the `capture_rotate_signals_internally == true` option in Async_file_logger
157 * constructor; the procedure will occur on receiving the proper signal automatically.
158 *
159 * @todo `Async_file_logger::log_flush_and_reopen(true)` is great for flushing, such as in an abort-signal handler,
160 * but providing just the flushing part without the reopening might be useful. At the moment we've left it
161 * this way, due to the vague feeling that closing the file upon flushing it is somehow more final and thus safer
162 * (in terms of accomplishing its goal) in an abort-signal scenario. Feelings aren't very scientific though.
163 *
164 * @param async
165 * If `true`, the operation will execute ASAP but asynchronously, the method exiting immediately;
166 * else it will complete fully before this method returns.
167 */
168 void log_flush_and_reopen(bool async = true);

204 /**
205 * This is the `Logger` doing all the real log-writing work (the one stored in Log_context is the
206 * logger-about-logging, `backup_logger_ptr` from ctor args). In itself it's a fully functional Logger, but
207 * its main limitation is all calls between construction and destruction must be performed non-concurrently --
208 * for example from a single thread or boost.asio "strand." The way we leverage it is we simply only make
209 * log-file-related calls (especially Serial_file_logger::do_log() but also
210 * Serial_file_logger::log_flush_and_reopen()) from within #m_async_worker-posted tasks (in other words from
211 * the thread #m_async_worker starts at construction).
212 *
213 * ### Design rationale ###
214 * What's the point of Serial_file_logger? Why not just have all those data and logic (storing the `ofstream`, etc.)
215 * directly in Async_file_logger? Answer: That's how it was originally. The cumulative amount of state and logic
216 * is the same; and there's no real usefulness for Serial_file_logger as a stand-alone Logger for general users;
217 * so in and of itself splitting it out only adds a bit of coding overhead but is otherwise the same. So why do it?
218 * Answer:
219 *
220 * The reason we split it off was the following: We wanted to sometimes log to the real file from within the
221 * #m_async_worker thread; namely to mark with a pair of log messages that a file has been rotated. It was possible
222 * to do this via direct calls to a would-be `impl_do_log()` (which is do_log() but already from the worker thread),
223 * but then one couldn't use the standard `FLOW_LOG_*()` statements to do it; this was both inconvenient and difficult
224 * to maintain over time (we predicted). At that point it made sense that really Async_file_logger *is*
225 * a would-be Serial_file_logger (one that works from one thread) with the creation of the one thread -- and other
226 * such goodies -- added on top. With that, `FLOW_LOG_SET_LOGGER(m_serial_logger)` + `FLOW_LOG_INFO()` trivially logs
227 * to the file (as long as it's done from the worker thread), accomplishing that goal; and furthermore it makes
228 * logical sense in terms of overall design. The latter fact I (ygoldfel) would say is the best justification; and
229 * the desire for `*this` to log to the file, itself, was the triggering use case.
230 */
231 boost::movelib::unique_ptr<Serial_file_logger> m_serial_logger;
232
233 /**
234 * The thread (1-thread pool, technically) in charge of all #m_serial_logger I/O operations including writes
235 * to file. Thread starts at construction of `*this`; ends (and is synchronously joined) at
236 * destruction of `*this` (by means of delegating to #m_async_worker destructor).
237 */
238 async::Single_thread_task_loop m_async_worker;
239
240 /**
241 * Signal set which we may or may not be using to trap SIGHUP in order to auto-fire
242 * `m_serial_logger->log_flush_and_reopen()` in #m_async_worker. `add()` is called on it at init iff [sic]
243 * that feature is enabled by the user via ctor arg `capture_rotate_signals_internally`. Otherwise this object
244 * just does nothing `*this` lifetime.
245 */
246 Signal_set m_signal_set;
