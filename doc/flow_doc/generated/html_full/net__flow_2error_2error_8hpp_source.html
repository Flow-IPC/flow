23/**
24 * Namespace containing the net_flow module's extension of boost.system error conventions, so that Flow network
25 * protocol API can return codes/messages from within its own new set of error codes/messages.
26 *
27 * ### Synopsis ###
28 *
29 * ~~~
30 * // Assign a code from net_flow's custom error code set -- to a *general* Error_code.
31 * flow::Error_code code = flow::net_flow::error::Code::S_CONN_REFUSED;
32 * // code contains the int equivalent of S_CONN_REFUSED enum value; and a category pointer for your new
33 * // new code set to which S_CONN_REFUSED belonds. So it's about the size of errno plus a pointer.
34 *
35 * // Yet we have nice OO access to the equivalents of `errno` and `strerror(errno)`:
36 * std::cout << "General error value = [" << code.value() << "]; msg = [" << code.message() << "].\n";
37 * // And we can throw an std::exception (`catch{}` can log the exception's .what() which will contain
38 * // the numeric value, the human message, and the optional context string).
39 * throw flow::error::Runtime_error(code, "Additional context info here!");
40 * ~~~
41 *
42 * ### Discussion ###
43 * The contents of this `namespace` is one instance (and model example) for how one
44 * can easily create their own simple error code set `enum` (and corresponding string messages) and hook it into
45 * the flexible, yet maximally fast, boost.system error facility. Another example (a peer of ours) would be
46 * boost.asio's error code set. See flow::Error_code doc header for a top-down summary of the
47 * Flow-used-and-recommended error reporting system.
48 *
49 * For boost.system experts wondering whether there is support for `error_condition` equivalence (i.e., the
50 * ability to compare a given `flow::error::error_code` to a more general `boost::system::error_condition`),
51 * currently the answer is no, there isn't. We may add `error_condition`s as #Error_code usage patterns emerge
52 * over time.
53 *
54 * @internal
55 *
56 * ### Summary of implementation ###
57 * This summarizes how we got the Synopsis above to work. Replicate this in other modules as desired.
58 * -# Define new code set `enum` in your own namespace (recommend `error` sub-namespace of your module) in
59 * a new header file (recommend name based on namespace name). Say we call it E.hpp.
60 * -# Also in E.hpp: Declare `make_error_code()` prototype right next to it.
61 * -# Also in E.hpp: Define `is_error_code_enum<>` specialization in `boost::system` namespace.
62 * -# In E.cpp: Define a new, non-public, category class C,
63 * whose main part is a simple function mapping each `enum` value to its human message (`switch()`).
64 * Instantiate a single `static` instance of C, perhaps as a member: `C::S_C`.
65 * -# Also in E.cpp: Implement the tiny boiler-plate `make_error_code()` from step 2.
66 * It constructs a general `Error_code` with the `int` version of the custom-`enum`
67 * value passed to it; and with a reference to `C::S_C` from previous step, thus identifying both the
68 * code set and the individual code.
69 *
70 * Result: One can directly assign codes from your new `enum` to the general `Error_code`
71 * (a/k/a `boost::system::error_code`); see Synopsis above.
72 *
73 * ### Implementation discussion ###
74 * While *using* this error reporting system is simple and convenient, the internal implementation is
75 * (IMO) cryptic albeit fairly concise. I followed the guide:
76 *
77 * http://blog.think-async.com/2010/04/system-error-support-in-c0x-part-4.html
78 *
79 * Note that this isn't some random web page but rather a detailed a walkthrough by the author of
80 * boost.system (which includes `error_code`) itself. In addition, its author wrote boost.asio, which we use
81 * extensively in this implementation. Finally, boost.asio's own extension of the `error_code`/etc. system --
82 * for its own error reporting, which is API-wise quite similar to this one -- can be seen (in boost.asio's
83 * largely headers-only implementation) to follow the above guide ~99% to the letter.
84 *
85 * What's rather odd is that boost.system's actual formal documentation, e.g.,
86 *
87 * http://www.boost.org/doc/libs/1_50_0/libs/system/doc/index.html,
88 *
89 * does not in any way explain how to implement all this properly. The above guide is not included in that
90 * official doc, nor is it linked. In fact, following the formal documentation without a supernatural (IMO)
91 * level of understanding of the author's intentions, would likely lead to the wrong implementation.
92 * Most visibly, for example, the docs say that `error_code` is for platform-dependent error codes; while
93 * `error_condition` is for portable error codes, or something very, very close to that.
94 * Well (I thought originally), Flow's errors ARE platform-independent, so I guess I should use
95 * `error_condition`s then! So originally, all the Flow APIs that reported errors took `Error_condition*`
96 * arguments, not `Error_code*` arguments. Only after some colleagues remarked that this appears wrong,
97 * and pointed out the above guide, did I realize that I did that wrong and switched to `Error_code`.
98 * In particular, the guide elucidates: `error_code` is to report a "low-level" or "system-level" specific
99 * error (not necessarily computer platform-dependent, as the formal doc might seem to say), while
100 * `error_condition` is something to (optionally) "check against" as a higher-level or more abstract piece
101 * of error handling logic. So in fact any original error should be an `error_code`, and `error_condition`
102 * is an _optional_ layer one _might_ add to help working with `error_code`s.
103 */

109/**
110 * All possible errors returned (via flow::Error_code arguments) by flow::net_flow functions/methods.
111 * These values are convertible to flow::Error_code (a/k/a `boost::system::error_code`) and thus
112 * extend the set of errors that flow::Error_code can represent.
113 *
114 * @internal
115 *
116 * When you add a value to this `enum`, also add its description to
117 * net_flow_error.cppâ€™s Category::message(). This description must be identical to the description
118 * in the /// comment below, or at least as close as possible.
119 *
120 * If, when adding a new revision of the code, you add a value to this `enum`, add it to the end.
121 * If, when adding a new revision of the code, you deprecate a value in this `enum`, do not delete
122 * it from this `enum`. Instead mark it as deprecated here and then remove it from
123 * Flow_category::message().
124 *
125 * Errors that indicate apparent logic bugs (in other words, assertions that we were too afraid
126 * to write as actual `assert()`s) should be prefixed with `S_INTERNAL_ERROR_`, and their messages
127 * should also indicate "Internal error: ".
128 *
129 * @todo Consider adding specific subclasses of flow::error::Runtime_error
130 * for the key codes, like `WAIT_INTERRUPTED` and `USER_TIMEOUT`.
131 * In doing so, make sure there's an automated syntactic-sugary way of doing this, so that one need only add
132 * a code<->class pair into some internal centralized mapping, automatically making system emit the proper
133 * exception class, defaulting to plain Runtime_error if not specially-mapped.
134 */

146 * Internal error: Tried to return Flow port which had not been reserved.
147 *
148 * @internal
149 *
150 * @note By its definition (see above brief description), it can only be triggered by Port_space::return_port().
151 * If one looks at all the calls to said method in practice, they'll note
152 * that it returning a truthy `*err_code` of any kind is unexpected to the point where it is an

213/**
214 * Given a `Code` `enum` value, creates a lightweight flow::Error_code (a/k/a boost.system `error_code`)
215 * representing that error. This is needed to make the
216 * `boost::system::error_code::error_code<Code>()` template implementation work. Or, slightly more in English,
217 * it glues the (completely general) flow::Error_code to the (net_flow-specific) error code set
218 * flow::net_flow::error::Code, so that one can implicitly covert from the latter to the former.

221 * The `enum` value.
222 * @return A corresponding flow::Error_code.

234/**
235 * Ummm -- it specializes this `struct` to -- look -- the end result is boost.system uses this as
236 * authorization to make `enum` `Code` convertible to `Error_code`. The non-specialized
237 * version of this sets "value" to false, so that random arbitary `enum`s can't just be used as
238 * `Error_code`s. Note that this is the offical way to accomplish that, as (confusingly but
239 * formally) documented in boost.system docs.
240 */
241template<>
242struct is_error_code_enum<::flow::net_flow::error::Code>
243{
244 /// Means `Code` `enum` values can be used for flow::Error_code.
245 static const bool value = true;
246 /* ^-- Could instead derive publicly from true_type, but then we'd have to find a true_type
247 * in some header we could use. At any rate this works just the same, and boost.asio does it this way, so if
248 * it's good enough for them (the author of boost.asio = the author of boost.system [error_code, etc.]),
249 * I say it's good enough for us. */
250};
